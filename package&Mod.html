Package y Modificadores de Acceso

Para que se usan los Package

En Java, los "paquetes" (packages en inglés) se utilizan principalmente para organizar y estructurar el código fuente en un proyecto. Los paquetes son directorios en los que se almacenan las clases relacionadas y otros recursos, y se utilizan por varias razones importantes:

Organización: Los paquetes permiten organizar las clases en una estructura jerárquica. Esto facilita la gestión de proyectos más grandes, ya que puedes agrupar clases relacionadas en paquetes y navegar por la estructura de directorios de manera más ordenada.

Evitar Conflictos de Nombres: Los paquetes ayudan a prevenir conflictos de nombres entre clases. Si tienes múltiples clases con el mismo nombre en diferentes paquetes, no habrá conflictos, ya que el nombre completo de una clase incluye el nombre del paquete.

Acceso Controlado: Los paquetes proporcionan niveles de acceso controlado. Las clases en un paquete pueden tener acceso a clases y miembros (atributos y métodos) marcados como "package-private" (sin modificador de acceso), lo que significa que solo las clases del mismo paquete pueden acceder a ellos.

Reutilización de Código: Los paquetes también fomentan la reutilización de código al organizar clases relacionadas en bibliotecas o módulos que pueden ser utilizados en múltiples proyectos.

Distribución de Bibliotecas: Cuando creas bibliotecas o componentes reutilizables en Java, a menudo los empaquetas en paquetes para que puedan ser distribuidos y utilizados por otros desarrolladores.

Para declarar que una clase pertenece a un paquete específico en Java, se utiliza la declaración package al principio del archivo de código fuente, antes de la declaración de la clase. Por ejemplo:

package com.ejemplo.miproyecto.mipaquete; public class MiClase { // Código de la clase }

El código anterior declara que la clase MiClase pertenece al paquete com.ejemplo.miproyecto.mipaquete. Esto ayuda a organizar las clases en una jerarquía y a evitar conflictos de nombres con otras clases.

Convención de nombres:

El estándar de nombrado de paquetes en Java no está estrictamente definido por el lenguaje en sí, pero existen convenciones ampliamente aceptadas y recomendadas por la comunidad de desarrollo de Java. Estas convenciones ayudan a que el código sea más legible y a que los desarrolladores puedan ubicar clases y paquetes de manera más fácil. Algunas de las convenciones comunes para dar nombre a los paquetes en Java incluyen:

  • Convenio de Nombres en Minúsculas: Los nombres de paquetes suelen escribirse en minúsculas. Por ejemplo, "com.ejemplo.miproyecto.mipaquete".
  • Jerarquía de Nombres Reversa de Dominio: A menudo, se comienza el nombre del paquete con el dominio inverso del sitio web de la organización o con el nombre de la empresa. Esto ayuda a garantizar que los nombres de paquetes sean únicos y evita colisiones. Por ejemplo, "com.ejemplo.miproyecto".
  • Nombres Descriptivos: Los nombres de los paquetes deben ser descriptivos y reflejar el propósito o la funcionalidad de las clases contenidas en ellos. Por ejemplo, "com.ejemplo.miproyecto.mipaquete.util" para paquetes de utilidades.
  • Evitar Abreviaciones Confusas: Evita abreviaciones o acrónimos oscuros que puedan ser confusos. Es preferible un nombre largo pero descriptivo a un nombre corto e incomprensible.
  • No Usar Palabras Reservadas de Java: No uses palabras reservadas del lenguaje Java, como "java" o "javax", como parte de un nombre de paquete, ya que pueden dar lugar a conflictos y problemas.
  • Utilizar Convenciones de Dominio: Si tu organización o proyecto tiene convenciones específicas de nombres de paquetes, es importante seguirlas para mantener la consistencia en todo el código.

Es importante recordar que estas son convenciones y no reglas estrictas del lenguaje Java. Sin embargo, seguir estas convenciones ayuda a que el código sea más fácil de entender y mantener, y fomenta una mayor consistencia en el desarrollo de software en Java.

Import y modificadores de acceso.

Lo primero que convien define el un programa y en la creación de clases, interfaces y enum es el paquete dónde se encuentra:

package com.ejemplo.miproyecto.mipaquete;

Para utilizar clases que se encuentran dentro de otro package necesitamos importar dichas clases y que sus modificadores de acceso lo permitan.

Las importanciones se realizan así:

import es.felix.mypaquete.MyClase;

Si queremos importar todas las clases que se encuentren dentro de un package se utiliza el elemento comodín.

import es.felix.mypaquete.*;

Si se importar dos clases con el mismo nombre en un mismo archivo sólo una de ellas podremos utilizarla con su nombre, el resto tenemos que utilizar el nombre completo

import es.felix.mypaqueteprimero.ClaseEspejo; import es.felix.mypaquetesegundo.ClaseEspejo; public class principal { main(){ //Equivale a la clase del primer paquete ClaseEspejo myclass1 = new ClaseEspejo(); //Equivale a la clase del segundo paquete ClaseEspejo myclass2 = new es.felix.mypaquetesegundo.ClaseEspejo() } }

Si dos clases se encuentran en el mismo paquete no se necesita importar nada.

Modificadores de acceso

  • default: Sólo se puede acceder desde el mismo package
  • protected: Se puede acceder desde el mismo package o cualquier otra clase que extienda de esta.
  • private: Sólo se puede acceder desde la misma clase
  • public: Se puede acceder desde cualquier clase.
